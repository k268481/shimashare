# シマシェア（ShimaShare）仕様書

## 1. サービス概要

### 1.1 ミッション
台風被害を受けやすい日本の離島において、平常時は物資シェアリング・店舗取り寄せ仲介のコミュニティアプリとして、緊急時は住民同士の助け合いインフラとして機能する**二面性**を持つLINEミニアプリ。

### 1.2 サービス名
- 正式名称: シマシェア
- 英語表記: ShimaShare

### 1.3 形態
- LINEミニアプリ（LIFF: LINE Front-end Framework）
- 将来的にiOS/Androidネイティブアプリへの展開を視野（フェーズ3以降）

### 1.4 対象エリア
- MVP: 沖縄・九州・瀬戸内のうち、台風被害を受けやすい**特定の1島**をパイロット地域とする
- 多島展開は当面行わず、**1島専用設計**でシンプルさを優先する
- パイロット地域は別途協議のうえ確定する

### 1.5 ターゲットユーザー
- 離島の住民（10代後半〜高齢者まで）
- 島内に登録される個人商店・小売店
- ユーザー登録は18歳以上、ただし同世帯家族による代理操作を認める

---

## 2. 提供機能の全体像

シマシェアは以下2つの機能を平常時から並行提供する。

### 2.1 個人間シェアリング機能（フリマ型）
住民同士が物資を出品し、取引相手とつながる。

### 2.2 店舗取り寄せ仲介機能（プラットフォーム型）
島内の複数の店舗に対し、住民が「取り寄せリクエスト」を出して店舗が応答する。

### 2.3 緊急（台風）モード
気象APIに連動して自動的にアプリの挙動・UIが切り替わり、助け合いを促進する。

## 3. 個人間シェアリング機能

### 3.1 出品

| 項目 | 内容 |
|------|------|
| 値段設定 | 出品者が自由に設定（0円〜上限なし） |
| 写真 | 最大4枚アップロード可（必須1枚） |
| 説明文 | 任意・最大500文字 |
| カテゴリ | 必須選択（6大分類、3.6項参照） |
| 受け渡し方法 | テキストで自由記入（例：「水曜の夕方 上嶺地区集会所近くで」） |
| 配送 | アプリ機能としては取り扱わない（受け渡し方法を出品者・購入者間で取り決め） |

### 3.2 マッチング（チャット）
- 取引希望者が出品にアプローチすると、1対1チャットルームが開設される
- チャット開始時点では双方の表示は「ニックネーム＋地区名」のみ（ジモティー方式）
- 実名・住所・電話番号などは**取引成立後も自動開示しない**。当事者が必要に応じてチャット内で開示する設計
- チャットは画像送信に対応。位置情報共有は任意

### 3.3 取引フロー
1. 購入者がチャットで申し込み
2. 受け渡し日時・場所を当事者間で合意（アプリ外でもOK）
3. アプリ外で現金受け渡しを実施
4. 双方が「取引しました」ボタンを押下
5. 双方が3段階評価＋コメントを残す（任意。スキップ可）

### 3.4 取引完了の判定
- 双方の「取引しました」ボタンが押された時点で正式に完了表示
- 一方のみ押下の場合は「片側完了」表示で、相手に通知。7日経過で自動完了
- どちらも押さなかった場合は7日経過で「期限切れ」表示。後述のトラブル対応で運営に申し立て可能

### 3.5 評価システム
- 3段階評価：◎（よかった）／○（ふつう）／△（よくなかった）
- 任意コメント（最大200文字）
- プロフィールに直近10件と全体の評価分布を表示
- △評価が短期間に集中したアカウントはモデレーター画面で警告フラグ

### 3.6 カテゴリ設計（6大分類）
1. 食品（生鮮含む）
2. 日用品（洗剤・トイレットペーパー等）
3. 衣類
4. ベビー・キッズ用品
5. 家電・道具
6. その他

- 詳細タグは付与しない（MVPでは全文検索に依存）
- 台風モード中は「**台風備蓄**」「**避難所提供**」など台風専用タグでフィルタリング可能

### 3.7 禁止・制限品目
- **禁止**: 医薬品全般（薬機法対応）、生体動物、危険物、武器類、たばこ、出会い系を目的とした出品
- **要注意（警告表示・自己責任）**: 生鮮食品、自家製食品
- **店舗を介する場合のみ可**: 酒類（個人間は禁止）

### 3.8 「ゆずります」（0円出品）
- 平常時から利用可能だが、台風モード中はホーム画面に専用バナーで前面表示
- 報酬を受け取らないため、決済法上のトラブルリスクが低く、災害時に重要

## 4. 店舗取り寄せ仲介機能

### 4.1 店舗の運用方針
- 在庫管理は要求しない（**リクエスト応答型**）
- 店舗は事前登録不要のリスト（「取り扱い品目」自由記入）のみ保持
- 住民がリクエストを送信した際、店舗が「OK／NG／代替提案」を返す

### 4.2 通知チャネル（ハイブリッド）
店舗は登録時に好みのチャネルを選択：

| チャネル | 仕組み |
|----------|--------|
| LINE | シマシェア公式アカウント1つのBotが各店舗のLINEにメッセージ送信。返信はOK/NGクイックリプライ |
| 電話 | 住民は店舗の電話番号タップで直接架電。アプリ上の応答は不要 |

- LINE経由を**第一推奨**とし、デジタル受信に抵抗のある店舗のために電話オプションを残す
- 店舗ごとに切替可

### 4.3 リクエスト内容
| 項目 | 内容 |
|------|------|
| 品目 | テキスト入力（例：5kg米、子ども用紙オムツMサイズ） |
| 数量 | 数値 |
| 希望日時 | 任意 |
| 備考 | 任意・最大300文字 |

### 4.4 店舗側UI（LINE Bot）
- メッセージ例：「【シマシェア】上嶺地区のしまちゃんから取り寄せリクエスト：5kg米×1、希望：今週水曜まで。応答してください→ ［OK／NG／代替提案］」
- クイックリプライボタンで応答 → アプリ側に反映
- 「代替提案」を選択するとフリーテキスト入力モードへ

### 4.5 取引完了
- 個人間取引と同じく、住民が「受け取りました」ボタン押下で完了
- 店舗側の評価機能は持たない（個人事業主への直接的な☆評価は摩擦が大きいため）

## 5. 緊急（台風）モード

### 5.1 起動トリガー
- **気象庁API（または同等の気象データAPI）と連携し、完全自動切り替え**
- 対象地域に「暴風警報」または相当の警報が発令された時点でON
- 解除でOFF
- スーパー管理者は管理画面から手動で**強制ON / 強制OFF**を上書きできる（誤検知・テスト用）

### 5.2 通常モードとの差分（UI・挙動）

| 項目 | 通常モード | 緊急モード |
|------|-----------|-----------|
| 配色 | ターコイズ＋サンゴの離島カラー | やや赤・黄寄りの注意喚起カラー（色覚多様性に配慮しコントラストで差別化） |
| ホーム画面 | 出品一覧 | 「台風備蓄リクエスト」「ゆずります（0円）」「店舗取り寄せ」を上部に固定表示 |
| 通知優先度 | 通常 | プッシュ通知をすべて優先扱い |
| 手数料 | 通常通り徴収（フェーズ2以降） | **通常通り徴収（透明性重視。例外は設けない）** |
| 「ゆずります」バナー | 控えめ | 前面表示 |

### 5.3 緊急モード独自機能（MVPに含める）

#### 5.3.1 「これがない」ストック警告掲示板
- 住民が「○○がスーパー○○で品切れ」「水がコンビニ△△に入荷」を投稿できる掲示板
- 投稿はリアルタイム表示。古い投稿（24時間以上）は自動で薄表示
- 1投稿につき：地区名／店舗名（任意）／品目／状態（品切れ／入荷／在庫少）／コメント

#### 5.3.2 「ゆずります」前面表示
- 緊急モード中のホーム画面上部に「困っている人へ。0円で出品する」CTAボタン
- ワンタップで0円出品フォームに直接遷移
- 0円出品はカテゴリの先頭に固定表示される

### 5.4 オフライン対応方針
- **オンライン前提設計**（MVPではオフラインキャッシュは実装しない）
- ネットワーク切断時は「現在オフラインです。最終更新：YYYY/MM/DD HH:MM」と明示
- 偽の在庫情報・古い掲示を「最新」と誤認させないことを最優先

## 6. 信頼・安全性

### 6.1 本人確認
- **自己申告のみ**（MVP）
- メールアドレス（一意）+ パスワードで登録・ログイン。メールの一意性でアカウントのユニーク性を担保
- 氏名・住所・電話番号を登録（任意。運営側のみ閲覧可、他ユーザーには非開示）

### 6.2 個人情報の表示原則（ジモティー方式）
- 出品画面・チャット冒頭で表示するのは**ニックネームと地区名のみ**
- 本名・住所・電話番号は当事者間で必要に応じてチャットで開示
- 受け渡し場所も当事者間で自由に決定（アプリは関与しない）

### 6.3 トラブル対応
- 「コミュニティ規範を領ざす」スタンス（軽めの仲裁）
- 通報ボタンを各出品・チャットに設置
- 通報は現地モデレーターにエスカレーション → 必要に応じてスーパー管理者（事業者）が判断

### 6.4 アカウント停止
- △評価3件以上＋通報2件以上で自動的にモデレーターレビュー対象
- スーパー管理者のみがアカウント停止権限を持つ

### 6.5 利用規約のスタンス
- 取引のトラブルに関しては「当事者間解決を原則」
- ただしコミュニティ規範違反（嘘の投稿、誹謗中傷、迷惑行為）には運営が介入
- MVP期間中は「テスト中」を明示し、責任範囲を限定

---

## 7. 権限階層（3階層）

| ロール | 権限 |
|--------|------|
| **スーパー管理者** | 全機能、アカウント停止、緊急モード手動切替、店舗追加・編集、規約違反処理、データエクスポート |
| **現地モデレーター** | 通報レビュー、出品の非表示化、コミュニティ警告投稿、トラブル仲裁の窓口対応 |
| **一般ユーザー** | 出品、購入、取り寄せリクエスト、チャット、評価、通報 |

- 現地モデレーターは原則「島内の信頼できる人物」（区長・自治会経験者など）
- 3階層UIは管理画面側で表示権限を出し分け

## 8. 技術スタック

> **2026-06 方針転換**: LINEミニアプリ（LIFF）前提から、**メールアドレス登録によるWebアプリ**へ変更した。
> 理由：(1) まずローカル環境で各ユーザーが登録・ログインして使える状態を最優先したい、(2) 外部サービス（LINE審査・Supabaseクラウド）の契約・リードタイムに先行依存せず開発を進めたい。
> LINE LoginとSupabaseは将来の選択肢として残すが、MVPはこれらに依存しない設計とする。

### 8.1 採用技術（現行：ローカルWebアプリ版）

| 領域 | 技術 |
|------|------|
| フロントエンド | Next.js (App Router) + React + TypeScript |
| 認証 | **メールアドレス + パスワード**（自前セッション：JWT署名付きhttpOnly cookie、bcryptでハッシュ） |
| バックエンド | Next.js Server Actions / Route Handlers |
| データストア | **SQLite**（better-sqlite3、完全ローカル）。データアクセスは`lib/repos.ts`に集約しDBを抽象化 |
| 通知 | フェーズ5で Web Push（PWA） + メール（SMTP） |
| 気象データ | 気象庁公式API または Open-Meteo（フェーズ4） |
| 決済 | **MVPは未実装**。フェーズ2でStripe検討 |
| ホスティング | ローカル開発（将来 Vercel） |
| 開発環境 | Windows + WSL2 + Node.js 20 + npm |
| バージョン管理 | GitHub |

### 8.2 採用理由
- **SQLite + 自前認証**: Docker・外部サービス契約なしで、各ユーザーがメール登録・ログインして使えるWebアプリを即座にローカルで動かせる。データアクセスを`lib/repos.ts`に集約しているため、将来Supabase(PostgreSQL)へ移行しやすい
- **メール + パスワード認証**: LINE審査やLIFF設定を待たずに、誰でもメールアドレスで登録・ログインできる
- **Next.js Server Actions / Route Handlers**: 別途APIサーバーを立てずにフォーム送信・チャット更新を実装でき、ローカル開発がシンプル
- **MVP決済なし**: 資金決済業ライセンス問題を先送りし、開発速度と法的安全性を確保

### 8.3 将来の移行・非採用
- **Supabase (PostgreSQL)**: 本番でマルチユーザー・RLS・Realtimeが必要になった段階で移行候補。`supabase/migrations/`のPostgreSQLスキーマは参考として残す
- **LINE Login / LIFF**: 離島での普及状況を見て再検討。現行は不要
- Firebase: NoSQLでの取引集計の複雑さ、ロックイン懸念
- オフラインキャッシュ: MVPでは偽情報リスクを優先回避

## 9. デザイン方針

### 9.1 ブランドイメージ
**「離島の温かさ」**
- 海・太陽・フランクな雰囲気
- アクセントカラー：ターコイズ（海）＋サンゴ（夕焼け）
- 高齢者にも親しみやすく、よそよそしくない

### 9.2 UI原則（高齢者ファースト）
| 項目 | 規定 |
|------|------|
| フォントサイズ | 本文16px以上、ボタン文字18px以上 |
| ボタンサイズ | 最小タップ領域44×44px、CTAは56px |
| コントラスト | WCAG AA以上（4.5:1） |
| 画面遷移 | 直線的、戻るボタン常設 |
| 文言 | 平易な日本語、専門用語は併記（「出品＝売りに出す」） |
| アイコン | テキストラベルとセットで使用 |

### 9.3 台風モードの視覚変化
- ホームのヘッダーが赤・黄寄りに変化
- 「【台風モード】現在、暴風警報発令中」バナー常設
- 通常モードに戻る瞬間にも明確なフィードバック

## 15. リスクと対策（再掲・優先順位順）

| # | リスク | 対策 |
|---|--------|------|
| 1 | 出品者・購入者間のトラブル | 評価・通報・モデレーター仲裁。MVPは現金決済なので金銭トラブルは限定的 |
| 2 | 店舗が乗り気でない | LINE Bot方式で店舗側UI負担をゼロ化、初期は管理者代行も視野 |
| 3 | 決済システムの法的問題 | MVPでは現金決済のみ。フェーズ2で資金移動業／決済代行を選定 |
| 4 | 台風時のネット回線断絶 | オンライン前提＋最終更新時刻明示で偽情報を防ぐ |
| 5 | 高齢者のアプリ操作 | 同世帯代理利用許可、大ボタン・平易な日本語、現地モデレーター対面サポート |
| 6 | 個人情報漏洩 | ジモティー方式で原則非開示、Supabase RLSで権限制御 |
| 7 | 誤った台風モード起動 | 気象庁API＋管理者強制ON/OFF権限の二段構え |
| 8 | コミュニティ規範違反（誹謗中傷等） | 通報→モデレーター→アカウント停止のエスカレーションパス |
